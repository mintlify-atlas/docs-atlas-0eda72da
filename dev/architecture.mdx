## Overview

Patient Manager follows a **layered architecture** with clear separation of concerns:

```
┌─────────────────────────────────────────┐
│         Presentation Layer              │
│    (Swing GUI Components)               │
│  MainWindow, Forms, Panels, Dialogs     │
└───────────────┬─────────────────────────┘
                │
                ▼
┌─────────────────────────────────────────┐
│         Service Layer                   │
│    (Business Logic)                     │
│  PatientService, SessionService, etc.   │
└───────────────┬─────────────────────────┘
                │
                ▼
┌─────────────────────────────────────────┐
│         Repository Layer                │
│    (Data Access)                        │
│  PatientRepository, BaseRepository      │
└───────────────┬─────────────────────────┘
                │
                ▼
┌─────────────────────────────────────────┐
│         Persistence Layer               │
│    (JPA/Hibernate + SQLite)             │
│  EntityManager, JPAUtil                 │
└─────────────────────────────────────────┘
```

## Design Patterns

### 1. Repository Pattern

All data access is encapsulated in repository classes that extend `BaseRepository`.

**BaseRepository** (`BaseRepository.java:16-53`):

```java
public abstract class BaseRepository {
    protected EntityManagerFactory emf;

    protected BaseRepository(EntityManagerFactory emf) {
        this.emf = emf;
    }
    
    protected EntityManager getEntityManager() {
        return emf.createEntityManager();
    }

    protected <T> T execute(Function<EntityManager, T> action) {
        EntityManager em = getEntityManager();
        try {
            em.getTransaction().begin();
            T result = action.apply(em);
            em.getTransaction().commit();
            return result;
        } catch (RuntimeException ex) {
            if (em.getTransaction().isActive()) {
                em.getTransaction().rollback();
            }
            throw ex;
        } finally {
            em.close();
        }
    }
}
```

**Key Features**:
- Automatic transaction management
- Automatic rollback on errors
- Resource cleanup in finally block
- Generic execute method for all operations

**Example Repository** (`PatientRepository.java:15-61`):

```java
public class PatientRepository extends BaseRepository {
    public PatientRepository(EntityManagerFactory emf) {
        super(emf);
    }

    public Patient create(Patient patient) {
        return execute(em -> {
            em.persist(patient);
            return patient;
        });
    }

    public Patient update(Patient patient) {
        return execute(em -> em.merge(patient));
    }

    public Patient findById(Long id) {
        return execute(em -> em.find(Patient.class, id));
    }

    public List<Patient> findAll() {
        return execute(em ->
            em.createQuery("from Patient", Patient.class)
              .getResultList()
        );
    }
}
```

### 2. Service Layer Pattern

Business logic is isolated in service classes that coordinate repositories.

**PatientService** (`PatientService.java:16-39`):

```java
public class PatientService {
    private final PatientRepository patientRepository;

    public PatientService(EntityManagerFactory emf) {
        this.patientRepository = new PatientRepository(emf);
    }

    public Patient create(Patient p) {
        return patientRepository.create(p);
    }

    public Patient update(Patient p) {
        return patientRepository.update(p);
    }

    public Patient findById(Long id) {
        return patientRepository.findById(id);
    }

    public List<Patient> findAll() {
        return patientRepository.findAll();
    }
}
```

### 3. Service Manager (Facade Pattern)

The `ServiceManager` provides a single entry point to all services.

**ServiceManager** (`ServiceManager.java:13-54`):

```java
public class ServiceManager {
    EntityManagerFactory em = null;
    private PatientService patientService = null;
    private SessionService sessionService = null;
    private PatientRelativeRelationService patientRelativeRelationService = null;
    private RelativeObservationService relativeObservationService = null;
    private RelativeService relativeService = null;

    public ServiceManager(EntityManagerFactory em) {
        this.em = em;
        patientService = new PatientService(em);
        sessionService = new SessionService(em);
        patientRelativeRelationService = new PatientRelativeRelationService(em);
        relativeObservationService = new RelativeObservationService(em);
        relativeService = new RelativeService(em);
    }

    public PatientService getPatientService() {
        return patientService;
    }

    public SessionService getSessionService() {
        return sessionService;
    }
    
    // ... other getters
}
```

**Usage in GUI** (`MainWindow.java:26-28`):

```java
EntityManagerFactory em = JPAUtil.getEntityManagerFactory();
ServiceManager serviceManager = new ServiceManager(em);
```

### 4. Singleton Pattern

**JPAUtil** - Ensures single EntityManagerFactory instance:

```java
public class JPAUtil {
    private static EntityManagerFactory emf;

    public static EntityManagerFactory getEntityManagerFactory() {
        if (emf == null) {
            // Initialize once
            emf = Persistence.createEntityManagerFactory("patientPU", props);
        }
        return emf;
    }
}
```

**ThemeManager** - Global theme management:

```java
public final class ThemeManager {
    private static AppTheme current = AppTheme.CLASSIC;
    private static final List<Runnable> listeners = new ArrayList<>();

    private ThemeManager() {} // Prevent instantiation

    public static AppTheme getTheme() {
        return current;
    }

    public static void setTheme(AppTheme theme) {
        // Update theme and notify listeners
    }
}
```

### 5. Observer Pattern

The theme system uses observers to notify components of theme changes.

**Theme Listener Registration** (`ThemeManager.java:25-46`):

```java
public static void addThemeListener(Runnable r) {
    listeners.add(r);
}

public static void setTheme(AppTheme theme) {
    if(theme == current) return;
    
    current = theme;
    applyLookAndFeel(theme);
    
    // Notify all listeners
    for(Runnable r : listeners) {
        r.run();
    }
    
    refreshAllWindows();
}
```

**Component Registration** (`BackgroundPanel.java:31-35`):

```java
ThemeManager.addThemeListener(() -> {
    loadBackground();
    repaint();
});
```

### 6. Template Method Pattern

The `execute` method in `BaseRepository` is a template for database operations:

```java
protected <T> T execute(Function<EntityManager, T> action) {
    EntityManager em = getEntityManager();
    try {
        em.getTransaction().begin();      // 1. Start transaction
        T result = action.apply(em);      // 2. Execute custom logic
        em.getTransaction().commit();     // 3. Commit transaction
        return result;
    } catch (RuntimeException ex) {
        if (em.getTransaction().isActive()) {
            em.getTransaction().rollback(); // 4. Rollback on error
        }
        throw ex;
    } finally {
        em.close();                       // 5. Cleanup resources
    }
}
```

## Package Structure

### Model Package (`com.bo.patientmanager.model`)

Contains JPA entity classes representing the domain model:

- `Patient` - Patient information and demographics
- `ClinicalHistory` - Clinical session records
- `ClinicalHistoryFile` - Attachments to clinical records
- `Relative` - Patient relatives/contacts
- `PatientRelativeRelation` - Relationship between patients and relatives
- `RelativeObservation` - Notes about relatives
- `Session` - Therapy session information

### Repository Package (`com.bo.patientmanager.repository`)

Data access layer with repository classes:

- `BaseRepository` - Abstract base class with transaction management
- `PatientRepository` - Patient data access
- `ClinicalHistoryRepository` - Clinical history data access
- `RelativeRepository` - Relative data access
- `SessionRepository` - Session data access
- etc.

### Service Package (`com.bo.patientmanager.service`)

Business logic layer:

- `ServiceManager` - Facade for all services
- `PatientService` - Patient business logic
- `ClinicalHistoryService` - Clinical history operations
- `RelativeService` - Relative management
- `SessionService` - Session management
- etc.

## Benefits of This Architecture

### 1. Separation of Concerns
- UI logic separated from business logic
- Business logic separated from data access
- Easy to test each layer independently

### 2. Maintainability
- Changes to UI don't affect business logic
- Changes to database don't affect UI
- Clear location for each type of code

### 3. Reusability
- Services can be used by multiple UI components
- Repositories can be used by multiple services
- Custom components can be reused across views

### 4. Testability
- Each layer can be tested in isolation
- Mock dependencies for unit testing
- Integration tests can test full stack

### 5. Scalability
- Easy to add new entities (model → repository → service)
- Easy to add new UI views
- Easy to add new business logic
